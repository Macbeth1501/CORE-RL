Project Title: CORE-RL

Full Expansion: Cloud Optimization & Resource Efficiency Reinforcement Learning

Main Subtitle :

A high-fidelity OpenEnv benchmark for training autonomous AI agents to solve complex multi-objective Cloud FinOps challenges.

CORE-RL is a realistic OpenEnv environment that simulates a modern cloud resource dashboard. It challenges AI agents to act as FinOps engineers — finding and eliminating idle ("zombie") resources, right-sizing oversized instances, and strictly enforcing budget constraints without ever compromising business-critical services.

Functional Requirements
Real-world task simulation
The environment must simulate a task humans actually do. Not games, not toys. Examples: email triage, code review, data cleaning, scheduling, customer support, content moderation.
OpenEnv spec compliance
Implement the full OpenEnv interface: typed Observation, Action, and Reward Pydantic models. step(action) → returns observation, reward, done, info. reset() → returns initial observation. state() → returns current state. openenv.yaml with metadata. Tested via openenv validate.
Minimum 3 tasks with agent graders
Each task defines a concrete objective an agent must accomplish, with a programmatic grader that scores performance (0.0–1.0). Tasks should range: easy → medium → hard. Graders must have clear, deterministic success/failure criteria.
Meaningful reward function
Provides signal over the full trajectory (not just binary end-of-episode). Rewards partial progress toward task completion. Penalizes clearly undesirable behavior (e.g. infinite loops, destructive actions).
Baseline inference script
Uses the OpenAI API client to run a model against the environment. Reads API credentials from environment variables (OPENAI_API_KEY). Produces a reproducible baseline score on all 3 tasks.

Problem Description 

1. The Executive Summary
CloudControl-Env is a high-fidelity simulation of a modern multi-cloud billing and resource dashboard. It challenges AI agents to act as "FinOps Engineers"—autonomous auditors capable of navigating complex resource hierarchies to eliminate waste, right-size infrastructure, and enforce strict budgetary guardrails without compromising system availability.

2. The Problem (The "Why")
Enterprise cloud waste is estimated at over $30 Billion annually.
The Complexity Gap: Traditional "static" tools can flag waste, but they cannot act. Human engineers are too slow to manage thousands of ephemeral resources.
The Agentic Challenge: Current AI agents are good at coding but struggle with multi-objective reasoning (e.g., "I need to save money, but I must not delete the database").
The Benchmarking Gap: There is currently no standard OpenEnv environment that simulates the high-stakes, high-data density of a cloud billing console.

3. The Environment Logic
The environment tracks a state of CloudResources. Each resource has:
id: Unique identifier (e.g., srv-prod-01).
type: VM, Managed Database, or Storage Bucket.
utilization: A float (0.0–1.0) representing real-time CPU/Memory usage.
cost_per_hour: The financial weight of the resource.
is_critical: A boolean flag. If True, stopping this resource triggers an immediate failure.
Action Space:
stop_resource(id): Shuts down a resource (Cost becomes $0).
resize_resource(id, new_tier): Changes the tier (e.g., xlarge → micro), reducing cost but affecting performance limits.
no_op(): Do nothing for this step.

4. The 3-Tier Task System
Your environment will programmatically generate three distinct scenarios:
Task
Level
Objective
Success Criteria
"The Ghost Hunter"
Easy
Find and terminate "Zombie" resources (0% utilization).
100% of unused resources stopped.
"The Scale-Down"
Medium
"Right-size" a cluster where 5+ VMs are oversized (e.g., 5% usage on an 8-core machine).
Achieve >30% savings while maintaining health.
"The Budget Breach"
Hard
A "Hard Budget" of $500 is set. Total spend is $1,500.
Meet budget by pruning non-critical nodes only.


5. The "Dense" Reward Function
To ensure the agent learns effectively, we use a Multi-Component Reward System ($\sum R$):
Efficiency Reward ($+R$): * +0.2 for every $10% reduction in total hourly spend.
+0.3 "Surgical Bonus" if the agent targets a resource with $<5\%$ utilization.
Safety Penalty ($-R$): * -1.0 (Immediate Terminal Failure) if the agent calls stop_resource() on an is_critical: True resource.
Stability Penalty ($-R$):
-0.1 per step if the "System Health" (based on remaining capacity) drops below a threshold.
Final Success Bonus ($+1.0$):
Awarded by the Grader only if the final state meets the Task Objective (e.g., Budget Met).

6. Why This Wins the Hackathon
Real-World Utility (30%): Directly addresses a massive corporate pain point (Cloud Waste).
Novelty (10%): First of its kind in the OpenEnv registry.
Technical Depth: Requires the agent to parse structured data (Observations) and perform mathematical comparisons before acting.
Safety-First Design: Includes "Criticality" logic, which is the #1 concern for real-world AI deployment.

PROBLEM STATEMENT: CloudControl-Env
The Objective
Build a complete OpenEnv environment that simulates a Cloud FinOps (Financial Operations) dashboard. The environment must challenge an AI agent to act as an autonomous Cost Optimization Engineer, tasked with identifying and remediating cloud waste across a fleet of simulated resources (Virtual Machines, Databases, and Storage) without compromising system integrity.

The Problem (Real-World Utility)
Enterprise cloud waste is estimated at over $30 Billion annually. While traditional tools can flag "zombie" resources, they cannot autonomously take action. Modern AI agents are currently evaluated on games or coding tasks, but there is a significant gap in benchmarks for Agentic Resource Management—where an agent must balance mathematical optimization (saving money) with strict safety constraints (not deleting production databases).

Environment Specifications
1. Observation Space (State)
The agent receives a list of CloudResource objects, each containing:
Resource ID: Unique identifier (e.g., srv-prod-01).
Type: Compute, Database, or Storage.
Utilization: CPU and Memory usage percentages (0.0–1.0).
Cost: Hourly spend in USD.
Criticality: A boolean flag (is_critical). If True, stopping this resource triggers a system failure.
2. Action Space
The agent must choose from the following typed actions:
stop_resource(id): Terminates the resource (reduces cost to $0.00).
resize_resource(id, new_size): Scales a resource up or down (e.g., xlarge → micro).
no_op(): Maintains current state.
3. Tasks & Difficulty Levels
You must implement a programmatic Grader for these three tasks:
Easy (The Ghost Hunter): Identify and stop three "Zombie" resources with 0% utilization.
Medium (Right-Sizing): A cluster of 5 VMs is oversized (using <5% capacity). The agent must resize them to a lower tier to achieve a 40% cost reduction.
Hard (The Budget Breach): Total spend is $1,200/mo. The budget is $500. The agent must prune non-critical resources to meet the budget while ensuring all is_critical: True resources remain running.

The Reward Function
The environment must provide a dense reward signal to guide the agent:
Positive (+): Proportional to the percentage of USD saved per step.
Surgical Bonus: Extra points for targeting resources with the lowest utilization.
Negative (-): Small penalty for every step taken (to encourage efficiency).
Terminal Penalty (-1.0): Immediate failure if a critical resource is stopped or resized below its required capacity.

Technical Requirements
Compliance: Must pass openenv validate.
Baseline: Provide an inference.py script using the OpenAI client that attempts to solve these tasks.
Deployment: Dockerized environment hosted on Hugging Face Spaces.
Constraints: Must run on 2 vCPUs and 8GB RAM; total inference runtime < 20 minutes.
